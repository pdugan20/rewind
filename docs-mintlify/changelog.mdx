## MCP server v0.4.2 -- Instapaper deep-links + anti-hallucination + drift prevention

Three targeted improvements based on real-session testing of v0.4.1.

- **Instapaper deep-links on every reading article.** The reading API response now carries two extra URL fields:
  - `instapaper_url` — `https://www.instapaper.com/read/{bookmark_id}`, the paywall-free archive copy with your highlights
  - `instapaper_app_url` — `instapaper://read/{bookmark_id}`, iOS deep-link to the Instapaper app
  Flows through `/v1/reading/recent`, `/v1/reading/articles/{id}`, `/v1/reading/articles/{id}/related`, `/v1/reading/highlights/*`, and `/v1/search` reading hits. MCP tools emit all three link variants (source, Instapaper web, Instapaper app) as distinct `resource_link` blocks. Ask Claude "give me the Instapaper link for that" and it can pull the Instapaper URL straight from the tool response.
- **Anti-hallucination server instructions.** Testing surfaced Claude fabricating biographical details ("X was a writer on SNL") to bridge a vague query to a retrieved article when the excerpt didn't support the claim. New `SERVER_INSTRUCTIONS` enforce: only assert facts present in the excerpt, offer candidates when the top match doesn't clearly fit, quote short phrases to show sourcing, and fetch `@rewind://article/{id}` for the excerpt when in doubt. The `find-article` prompt gets a matching anti-hallucination block.
- **Explicit tool routing nudge.** `SERVER_INSTRUCTIONS` now explicitly tells the model to prefer Rewind tools over conversation search / web search when the user references their own history. Caught the routing case where Claude was reaching for its built-in conversation search instead of the Rewind MCP on "find that article I saved".
- **Drift prevention (infrastructure).** New `manifest-snapshot.test.ts` snapshots every registered tool / prompt / resource / URI template — adds or renames show up as a reviewable diff. New `scripts/check-docs.mjs` fails CI when a registered identifier is absent from the Mintlify docs. Tightened `server.test.ts:774` from `toBeGreaterThanOrEqual(29)` to `toBe(39)` — the inequality assertion was the reason `get_health` shipped undocumented in a prior release.

## MCP server v0.4.1 -- click-through links + cleaner output

Polish pass on the v0.4.0 reading-search shipment. No new capabilities; existing tools read better in chat and link out to the actual source.

- **External URLs on search results.** `search`, `semantic_search`, and `find_similar_articles` now emit the article's real URL (nytimes.com, etc.) as a `text/html` resource_link alongside the internal `rewind://` drill-in. Models and clients can present clickable links to the source, not just opaque IDs.
- **Inline domain + author.** Search result text lines now read `The Secret Weapon of S.N.L. by Jason Zinoman (nytimes.com)` instead of the FTS-internal `[reading/article]` prefix.
- **No more `[ID: 42]` noise.** Stripped the internal-id bracketed prefix from user-visible text across all list tools (`get_recent_reads`, `get_recent_runs`, `get_recent_watches`, `browse_movies`, `get_top_*`, `get_personal_records`, collecting browse tools). IDs still live in `structuredContent.items[i]` for model chaining — they just no longer clutter the prose.
- **Per-item drill-in links.** `browse_movies` and `get_personal_records` now emit `rewind://movie/{id}` and `rewind://activity/{id}` links per row.

## API

- **`/v1/search`** responses include `url` and `author` fields for reading-article hits (null for other domains). Semantic path fills them via the existing JOIN; FTS path does a single follow-up lookup in `reading_items`.

## MCP server v0.4.0 -- semantic recall for reading

Introduces meaning-based search over the reading domain. Ask about articles by what they were *about*, not just the words in them.

- **`semantic_search` tool** — Paraphrased / meaning-based recall via Voyage AI embeddings (voyage-3-lite, 512 dim, cosine). Returns articles ranked by similarity with a score in [0,1]. Reading domain only.
- **`search` tool gains `mode` param** — `keyword` (default, unchanged FTS behavior), `semantic`, or `hybrid`. Hybrid combines FTS + semantic via reciprocal rank fusion (k=60). Semantic and hybrid are reading-only.
- **`find_similar_articles(article_id)` tool** — Thematically related articles via cosine similarity against the stored article vector. No Voyage API call at query time — reuses the stored embedding.
- **`find-article` prompt** — Slash-command style template: "Find that article I vaguely remember about..." → runs hybrid search, falls back to pure semantic if needed, surfaces top match plus 3-5 related pieces.
- **`rewind://highlight/{id}` entity resource** — New drill-in URI for saved highlights; search hits on highlight text become clickable resource_links.

## API

- **`GET /v1/search?mode=keyword|semantic|hybrid`** — Cross-domain search now supports three ranking modes. Semantic/hybrid require `domain=reading` and return a cosine `score` alongside each result.
- **`GET /v1/reading/articles/{id}/related?limit=5`** — Related-articles endpoint backed by Cloudflare Vectorize.
- **`POST /v1/admin/reembed-reading`** — Admin backfill: embeds all reading items into the `rewind-reading` Vectorize index via Voyage. Idempotent.
- **`POST /v1/admin/reenrich-reading`** — Admin retry for Instapaper articles that previously failed enrichment (e.g. paywall 403s on the OG-fetch step). Rescues most by reading Instapaper's cached body text via `get_text`.
- **`POST /v1/admin/backfill-body-excerpt`** — Derives `body_excerpt` from existing `reading_items.content` so the FTS index can match on article body.
- **`POST /v1/admin/reindex-search`** — Rebuilds the FTS `search_index` from source tables (optional `{domains: [...]}` filter).
- **`GET /v1/reading/highlights/{id}`** — Highlight detail endpoint with nested parent-article context.
- **FTS tokenizer**: dotted acronyms like `S.N.L.`, `A.I.`, `U.S.A.` are now normalized at index and query time so `SNL` matches `S.N.L.` (and vice versa).
- **`/v1/search` results** now include an `image` object (cdn_url, thumbhash, dominant_color) when the matched entity has an image row.

</Update>

## MCP server v0.3.0

Rich responses, interactive UI, and new tools across all domains.

- **Interactive poster grid** for `get_recent_watches` via [MCP Apps](https://blog.modelcontextprotocol.io/posts/2026-01-26-mcp-apps/). Renders inline in Claude Desktop, Claude web, VS Code GitHub Copilot, and Goose with thumbhash placeholders, theme-aware cards, and clickable Letterboxd links. Other clients get the existing text + image + resource_link + structuredContent response unchanged.
- **7 new tools**: `get_listening_genres`, `get_running_years`, `get_watching_genres`, `get_watching_decades`, `get_watching_directors`, plus `browse_movies` filtering by director and year.
- **10 entity resources** exposed as `@`-mentionable URIs (`rewind://movie/{id}`, `rewind://album/{id}`, `rewind://artist/{id}`, `rewind://vinyl/{id}`, etc.) so clients can drill from a search hit straight into the full record.
- **Richer responses**: detail and list tools now return cover art / posters / artist imagery as `image` content blocks (top-N, default 5, disable with `include_images: false`). External platform URLs (Letterboxd, Strava, Discogs, Apple Music, Last.fm) come back as `resource_link` blocks. Every tool also returns `structuredContent` for reliable numeric reasoning.
- **3 new prompts**: `letterboxd-review-draft`, `training-report`, `film-diet`.
- Fixed duplicate entries in `get_recent_watches` and corrected star-rating display (0-5 scale).

## API

- All `/recent` endpoints raised `limit` cap from 20 to 50 and gained a `page` parameter for offset pagination.

</Update>

## MCP server

New [MCP server](/mcp-server) for connecting AI assistants to your Rewind data. Published as [`rewind-mcp-server`](https://www.npmjs.com/package/rewind-mcp-server) on npm.

- 30 tools across all five domains: listening, running, watching, collecting, reading
- Cross-domain search, unified feed, and on-this-day
- Built-in prompt templates for weekly summaries, year-in-review, and period comparisons
- Works with Claude Desktop, Claude Code, Claude iOS/web, and any MCP-compatible client
- Local mode via `npx` or remote mode via `mcp.rewind.rest`

</Update>

<Update label="March 31, 2026" tags={["Improvements"]}>

## Reading domain: delta sync

Instapaper sync now uses the `have` and `highlights` API parameters for delta syncing. Instead of re-fetching all bookmarks and highlights every cycle, the sync sends known bookmark hashes and highlight IDs so the API only returns new or changed data. Also handles `delete_ids` to remove bookmarks deleted from Instapaper.

Added `verify_credentials` method to the Instapaper client for credential health checks. Fixed the `/health/sync` endpoint to include the Reading domain.

</Update>

<Update label="March 22, 2026" tags={["New features"]}>

## Reading domain

New Reading domain powered by Instapaper. Track articles, reading progress, and highlights.

- `GET /reading/articles` -- browse and filter articles by status, domain, tag
- `GET /reading/currently-reading` -- articles in progress
- `GET /reading/highlights` -- highlighted passages across all articles
- `GET /reading/highlights/random` -- random highlight for daily quote
- `GET /reading/stats` -- reading statistics and streaks
- `GET /reading/year/{year}` -- year in review

Articles include metadata extracted from OG tags: author, site name, word count, estimated read time, and thumbnail images.

</Update>
